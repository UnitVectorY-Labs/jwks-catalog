# AGENTS.md

Key assumptions and non-obvious details for AI agents working on this codebase.

## OIDC Domain and Issuer Matching

The `openid-configuration` URL domain in `services.yaml` must match the `issuer` field returned by the OIDC discovery endpoint. The OIDC specification ([RFC 8414](https://datatracker.ietf.org/doc/html/rfc8414)) requires that the issuer identifier is the prefix of the discovery URL. When the crawled issuer differs from the configured domain, the `openid-configuration` URL should be updated to use the issuer's domain, and the original domain should be added as an alias.

## Aliases

The `aliases` field on a service entry is a list of alternative domain names through which the same OIDC configuration can be discovered. Aliases represent domains that serve the same OIDC configuration (same issuer and JWKS URI) but are not the canonical issuer domain. Only the bare domain name is stored in the YAML; the full `https://{domain}/.well-known/openid-configuration` URL is constructed at build time for display purposes.

## Crawl Data Source

Domain and issuer information is validated against crawl results from [jwks-observer](https://github.com/UnitVectorY-Labs/jwks-observer). The observer stores per-service data in `data/{service-id}/oidc.json`, which contains the raw OIDC discovery response including the `issuer` field. This crawl data is the source of truth for determining whether a domain mismatch or duplicate exists â€” the catalog build environment cannot call OIDC endpoints directly.

## Duplicate Consolidation

When two service entries share the same JWKS URI and the same issuer (e.g., `dropbox.com` and `www.dropbox.com`), they should be consolidated into a single entry. The canonical entry uses the issuer-matching domain for its `openid-configuration` URL, and the other domain is added as an alias. The duplicate entry is removed.

## Known Non-Standard Issuers

Some services return issuers that do not follow the OIDC specification:

- **FedEx**: Issuer includes port `:443` (e.g., `https://auth.fedex.com:443/am/oauth2`). Since port 443 is the default for HTTPS, this is treated as equivalent to the configured domain.
- **Porsche (API)**: Issuer is malformed and missing the `https://` scheme prefix.
- **Zscaler variants**: The OIDC endpoint is served at `login.*.net` but the issuer claims `zservice.*.net`. The OIDC configuration URL cannot be changed because the endpoint is only reachable on the `login` subdomain.

These cases are left as-is because the mismatch is a server-side configuration issue that cannot be resolved by changing the catalog entry.

## Validation Rules

The `go run main.go -validate` command checks:

- Required fields (`id`, `name`, `jwks_uri`) are present on every service.
- No duplicate `id`, `name`, `jwks_uri`, or `openid-configuration` values.
- Aliases are non-empty, unique across all services, and do not duplicate the service's own OIDC domain.

## Build Process

The static site is generated by running `go run main.go` with environment variables:

- `WEBSITE`: Base URL for the generated site (e.g., `https://jwks.unitvectory.com`).
- `JWKS_OBSERVER_PATH` (optional): Path to a local checkout of jwks-observer for key history data.

Output goes to the `output/` directory. The site uses HTMX for client-side navigation with server-rendered HTML snippets as fallback.
